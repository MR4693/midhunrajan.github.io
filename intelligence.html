Executive Summary

The management layer built on top of the operator layer.

The Manufacturing Operations Platform gave operators digital tools to log production. The Production Intelligence Platform turns that production data into management insight.

The Production Intelligence Platform aggregates live production data from 19 operators and 14 work centres into four management dashboards, giving directors and supervisors real-time visibility into shop floor activity, work order status, and labour efficiency — without specialist ERP access or end-of-shift reporting. Built as the second platform on the same SQL infrastructure as the Manufacturing Operations Platform, it demonstrates how a shared data layer turns operator clock-on data into executive intelligence with no additional integration cost.

02 · Business Challenge

What management couldn't see. What that cost.

Before this platform, management visibility into production performance depended on end-of-shift reports, supervisor estimates, and ERP queries that required specialist knowledge to run.

The Visibility Problem

Before this platform, production managers at Ludwick Precision had no live view of the shop floor. Work order status existed in MIE Trak ERP, but extracting meaningful information required specialist query knowledge that supervisors and management did not have. Floor activity was only knowable by physically walking the shop or waiting for verbal updates from operators.

  • No real-time view of which work orders were active, overdue, or stalled — ERP data required specialist query access and training to extract meaningfully
  • Operator activity could only be confirmed by physical presence on the shop floor — supervisors had no digital visibility of who was working on what from their desk
  • Labour efficiency against job estimates was not tracked in real time — no audit trail existed for individual operator performance during a shift, only aggregate weekly summaries

The Business Impact

Without live production visibility, operational decisions lagged behind reality. Overdue jobs accumulated without management awareness, capacity planning relied on verbal briefings rather than data, and identifying bottlenecks required manual floor investigation rather than dashboard interrogation.

  • Overdue job accumulation went undetected until a customer escalation — 82 jobs overdue at deployment, with no automated alert mechanism in place to surface risk earlier
  • Capacity planning and scheduling decisions were made without live data on which work centres were loaded, stalled, or idle — decisions relied on supervisor memory and verbal briefing
  • Labour utilisation analysis relied on supervisor estimates rather than actual clocked hours — no per-operator efficiency data was available against ERP job time estimates
Before

Production visibility required physical floor walks, verbal supervisor briefings, or MIE Trak ERP queries requiring specialist knowledge — none available to management in real time or from desk. Problems surfaced through customer complaints, supervisor escalations, or manual end-of-day review.

After

Live manufacturing dashboard showing 314 open work orders, overdue flags, work centre loads, operator sessions, and labour efficiency — accessible from any browser on the factory network, updated the moment operators clock in or out on the Manufacturing Operations Platform.

One data layer. Two platform perspectives.

The Production Intelligence Platform doesn't duplicate the MIE Trak integration. It reads from the same SQL Server data layer that the Manufacturing Operations Platform writes to — creating a unified data architecture for the entire digital transformation.

SQL Integration

Shared Data Layer Architecture

The Production Intelligence Platform queries the same SQL Server instance used by the Manufacturing Operations Platform. Views and reporting queries are built on top of the operational tables — no data duplication, no ETL pipeline required.

Backend

Flask / Python Backend

A dedicated Flask application serving all four dashboard views — separate from the Manufacturing Operations Platform but sharing the same Python stack and factory network deployment. Each view maps to a distinct Flask route that executes pre-compiled SQL views against the shared SQL Server instance and passes aggregated data to Jinja2 templates for server-side rendering. No REST API consumption — data is assembled server-side on every page load, keeping the architecture simple and observable.

This was a deliberate decision: adding authentication overhead for a management-only internal tool on a closed factory network introduced complexity without commensurate security benefit. Network access is the access control boundary.

Zero additional data entry. Zero delay.

Every metric on the Production Intelligence Platform dashboard is derived from operator actions on the Manufacturing Operations Platform. The moment an operator clocks out and enters a quantity, that data is available to management on the Intelligence Platform. No batch jobs. No end-of-day reports. No manual consolidation.

Pull architecture. No outbound data flow.

There are no scheduled reports, email notifications, or push alerts in the current deployment. The data flow is entirely pull-based: management opens the browser, Flask executes SQL views, and the current production state is rendered. All data flows inward — from MOP operator actions, through SQL, to the management browser. No data flows outward from the Intelligence Platform to external systems at this stage. This keeps the architecture simple, eliminates scheduled-task failure modes, and ensures the dashboard always shows the freshest available data — not a cached result from the last scheduled run.

09 · Engineering Decision Records

Why this. Not that.

Engineering Decision Records document the key technical choices made during the development of this platform. They demonstrate engineering judgement — not just implementation skill.

Engineering Decision Record

EDR-01 · Build vs. Buy — Custom Platform vs. Off-the-Shelf BI
Problem

Management needed production visibility. The most obvious path was to connect an established BI tool (Power BI, Tableau) to MIE Trak's database and use pre-built dashboards. Should we use a commercial BI platform, or build a custom web application?

Options Considered

1. Microsoft Power BI connected to MIE Trak SQL — rapid deployment, existing Microsoft licensing

2. Tableau — powerful visualisation, enterprise-grade

3. Custom Flask web application — full control, same stack as MOP

✓ Chosen Solution

Custom Flask application. The Intelligence Platform needed to read from the same SQL layer as MOP — not MIE Trak directly. This shared-data architecture was only possible through a custom solution. It also shares authentication, deployment infrastructure, and internal network access with MOP.

Trade-offs & Lessons

More initial development time vs. out-of-the-box BI capabilities. Accepted: the shared SQL layer was the architectural cornerstone — it eliminated an entire ERP integration and enabled both platforms to share a single data foundation. No commercial BI tool offered that architectural alignment. A BI tool would also have required separate licensing, vendor dependency for updates, and a new toolchain to learn and maintain — costs only justified if the reporting data lived in a separate, BI-friendly data source. Here, the data was already being structured for operational use by MOP; the intelligence layer was a natural extension of an architecture already in flight.

Engineering Decision Record

EDR-02 · Data Architecture — Shared SQL Layer vs. Separate Analytics Database
Problem

The Intelligence Platform needed access to production data written by MOP. Should it read directly from MOP's SQL tables, maintain a separate analytics database with an ETL pipeline, or query MIE Trak's API directly at report time?

Options Considered

1. Separate analytics database — ETL pipeline from MOP SQL to a reporting-only database

2. Direct MIE Trak API polling — bypass SQL, query ERP on every dashboard load

3. Shared SQL layer — build reporting views on top of MOP's operational tables

✓ Chosen Solution

Shared SQL Server layer with reporting-specific SQL views. MOP writes to operational tables; the Intelligence Platform reads through views that aggregate and join that data into management-facing formats. Single source of truth. No ETL pipeline. No data duplication.

Trade-offs & Lessons

Reporting queries run on the same SQL instance as operational writes — a risk if query volume grows significantly. Mitigated through SQL view design and indexing on join columns. At Ludwick Precision's current scale, query execution is well within acceptable latency. If volume scales, read replicas would be the next step — but the shared-layer architecture makes that transition straightforward.

Engineering Decision Record

EDR-03 · Frontend Technology — Server-Rendered Templates vs. SPA Framework
Problem

Should the management dashboard UI be built as a Single Page Application (SPA) with React or Vue for rich interactivity, or as a traditional server-rendered Flask application consistent with MOP's architecture?

Options Considered

1. React SPA with Flask REST API — rich client-side interactivity, client-server separation

2. Vue.js SPA — lighter than React, similar trade-offs

3. Flask Jinja2 server-rendered templates with Chart.js — consistent with MOP architecture

✓ Chosen Solution

Server-rendered Flask templates with Chart.js for visualisation. This matches the Manufacturing Operations Platform architecture exactly — same stack, same deployment model, same developer context. Single Python process serves both API and rendered views.

Trade-offs & Lessons

No client-side routing; full page loads on navigation. For a management dashboard with moderate interactivity requirements, this is an acceptable trade-off. The architectural consistency with MOP was worth more than the interactive features a SPA would add — a single engineer maintaining two platforms benefits significantly from a shared templating system, single deployment model, and unified debugging context. SPAs introduce a separate build toolchain, hydration complexity, and client-state management that adds no value when the audience is internal management refreshing a dashboard a handful of times per shift.

Engineering Decision Record

EDR-04 · ERP Query Strategy — Direct MIE Trak API vs. SQL Views
Problem

MIE Trak holds the authoritative production record. Should the Intelligence Platform query the MIE Trak REST API directly on each dashboard load, or leverage the SQL layer MOP already maintains?

Options Considered

1. Query MIE Trak REST API directly on every dashboard refresh

2. Build SQL views on the shared data layer populated by MOP

3. Scheduled extraction from MIE Trak into a dedicated reporting table

✓ Chosen Solution

SQL views on the shared data layer. MOP already writes relevant production events to SQL on every operator action. Building views on this layer gives the Intelligence Platform structured access without additional API calls, rate limit risk, or ERP dependency at render time.

Trade-offs & Lessons

The Intelligence Platform's data is only as current as MOP's SQL writes — it reflects operator-clocked production, not every ERP state change. For this use case (management visibility into shop floor activity), this is a feature: the dashboard shows exactly what operators have recorded, which is the ground truth for production reporting.

Engineering Decision Record

EDR-05 · Reporting Architecture — Real-Time Dashboard vs. Scheduled Batch Reports
Problem

The simplest intelligence implementation would be scheduled batch reports — PDF or email summaries at shift end. Should the platform deliver live dashboards or scheduled reports that require no continuous server load?

Options Considered

1. Scheduled batch reports — email/PDF at shift end or on demand

2. Live dashboard — SQL queries executed on page load, data always current

3. WebSocket push — server pushes updates as production events occur

✓ Chosen Solution

Live dashboard with query-on-load. Every page load executes the SQL aggregations and renders fresh production data. No scheduling infrastructure. No email delivery pipeline. Management sees data current as of the last operator clock-out — no delay, no batch window.

Trade-offs & Lessons

Each page load triggers SQL queries — a consideration if data volume grows significantly. At Ludwick Precision's current scale this is fast and unproblematic. The live approach was also the right product decision: a management intelligence platform that requires waiting for a scheduled email is a reporting system, not an intelligence platform. The distinction matters for adoption. Query performance was validated using SQL execution plans before each view went live — confirming that aggregation ran against pre-indexed views, not full table scans against live transaction data.

10 · Technical Challenges & Solutions

Real engineering problems. Real engineering solutions.

The production challenges encountered building this platform — and how each was resolved. These are the problems that don't appear in tutorials.

CHALLENGE 01
Problem

The Intelligence Platform's aggregation queries run on the same SQL Server instance as MOP's real-time operator operations. A slow reporting query at the wrong moment could block operator clock-in transactions — unacceptable in a production environment.

ENGINEERING SOLUTION
Solution

All reporting views were designed with pre-aggregated SQL views and appropriate indexing on the primary join columns (operator ID, work order number, timestamp). Execution plan analysis was run on each reporting query before deployment to confirm cost and identify table scan risks.

OUTCOME
Outcome

Reporting queries execute without impacting operator transaction performance. SQL view architecture ensures aggregation logic runs on pre-computed data rather than live transaction tables.

CHALLENGE 02
Problem

Work orders that are actively being worked on by operators are "in progress" — they have a clock-on session open but no completed timestamp. Standard reporting queries that aggregate completed job data would exclude these, making active jobs invisible on the dashboard.

ENGINEERING SOLUTION
Solution

SQL views were written to UNION both completed sessions (with end timestamps) and open sessions (where end timestamp IS NULL). The platform uses conditional logic to calculate elapsed time for open sessions using GETDATE() minus the open session start time, giving an accurate "time active so far" value.

OUTCOME
Outcome

Live Shop Floor and Employee Efficiency views show in-progress work in real time alongside completed sessions. Active sessions display live elapsed time — core to the supervisor use case.

CHALLENGE 03
Problem

Customer names, part numbers, and due dates live in the MIE Trak ERP database tables. The MOP clock-on data only stores work order numbers and operation IDs. To show a dashboard with customer name and due date alongside operator sessions, the platform needs to join across multiple ERP tables with correct key relationships.

ENGINEERING SOLUTION
Solution

SQL views were built that join MOP session data to MIE Trak's work order, sales order, customer, and part number tables using the shared work order number as the join key. Views were tested with known WOs to validate customer, part, and due date accuracy before being exposed to the dashboard layer.

OUTCOME
Outcome

Every dashboard view shows full context — customer name, part number, description, due date — alongside live operator and session data. The join architecture became the data foundation for all four platform views.

CHALLENGE 04
Problem

The platform is designed for management visibility — not for operators who are using the separate MOP system for their clocking workflow. Ensuring the platform is accessible on the factory network to managers and supervisors, while not disrupting operator workflows on the shop floor terminals, required a deliberate access approach.

ENGINEERING SOLUTION
Solution

The platform was deployed on the factory network as a browser-accessible application on a specific internal URL. Access is available from any device on the network — management laptops, office PCs, tablets — without requiring a separate login system. MOP and PIP operate as distinct applications serving distinct user groups.

OUTCOME
Outcome

Management accesses the platform from their desks or mobile devices without interrupting shop floor operations. The two-platform architecture cleanly separates operator-facing tools from management-facing intelligence.

12 · Lessons Learned

What building this platform taught about manufacturing systems.

Observations from designing and deploying a real-time intelligence layer on top of live ERP data in a production manufacturing environment.

Shared infrastructure demands careful query design

When reporting queries run on the same SQL Server instance as production transactions, query design is not optional — it's critical. Every aggregation view was reviewed against execution plans before deployment. The discipline required is significantly higher than in a dedicated analytics environment.

Management visibility requires different thinking than operator tools

MOP was designed for operators — fast, focused, action-oriented. PIP was designed for managers — broad, contextual, decision-enabling. The same data served two fundamentally different mental models. The UI design, information density, and refresh cadence all needed to be reconsidered for the management context.

Live data has a different contract than batch data

When a dashboard queries live production data, "incorrect" data is always possible — an operator clocked on to the wrong job, a work order not yet fully set up in MIE Trak. Handling NULL values, missing references, and edge cases gracefully is non-negotiable when the audience is management making real-time decisions.

The value is in the visibility, not the technology

The most valuable outcome of this platform was not the technical architecture — it was giving management a view they never had before. The SQL views, Flask routes, and Chart.js charts are implementation details. What matters is that a production manager can now see 82 overdue jobs, sorted by severity, from any device on the network, in real time.

13 · Key Takeaways

What this demonstrates.

The capabilities demonstrated by this platform in a professional context — for anyone evaluating this case study.

01

ERP integration without the ERP API

MIE Trak does not expose a developer-friendly REST API. The data access strategy — reading directly from the SQL Server database through parameterised views — achieved full ERP integration without API licensing, without vendor dependency, and with complete control over query design and performance.

02

One data layer, two platforms

The Manufacturing Operations Platform writes production data. The Production Intelligence Platform reads it. This architecture was deliberate: the MOP established the data foundation, and PIP consumes it. Building the intelligence layer on the same SQL infrastructure avoided data duplication, synchronisation problems, and additional infrastructure cost.

03

Manufacturing systems engineering at the application layer

The technical decisions in this platform — shared SQL, pre-aggregated views, live query architecture, multi-view dashboard design — reflect manufacturing systems engineering knowledge as much as software engineering knowledge. Understanding the production environment, the operator workflows, and the management information requirements was prerequisite to every technical choice.

04

Four production views replacing manual processes

Each of the four platform views replaces a manual process that previously required physical presence, ERP queries, or supervisor estimates. The Manufacturing Dashboard replaces the end-of-shift report. Live Shop Floor replaces the floor walk. Employee Efficiency replaces the headcount register. Production Insights replaces the weekly management report.
