Transformation Philosophy

Five principles that guide every system built

Digital transformation in manufacturing fails not from technical shortcomings, but from systems built in isolation from the floor. These principles reflect lessons from real deployments.

01

Production first, technology second

Every system starts with a manufacturing problem. If it doesn't improve something for operators, engineers, or management — it doesn't get built.

02

Integration over isolation

Standalone tools create new silos. Every new capability connects to the ERP, to existing data, to the people who use it every day.

03

AI augments; it does not replace

AI extends the capability of engineers and operators — it does not remove decision-making from the people who understand the process.

04

Ship early, improve continuously

A working system used imperfectly delivers more value than a perfect system that never reaches the floor.

05

Transparent about how it was built

Modern engineering means directing AI effectively — selecting tools, validating outputs, taking full responsibility for what is deployed.

Project Chapter · Phase 1

ERP Integration & SQL Data Platform

MIE TrakSQL ServerREST APIPython
Why it was built

No programmatic access to ERP data existed. Every downstream system would require a reliable, structured data layer — this was built first, intentionally.

Who benefited

Every subsequent system in the ecosystem. Without this foundation, none of the phases that followed would have been possible.

Operational problem solved

Eliminated the manual data extraction that made reporting delayed and inaccurate. Created a live, queryable data layer across the entire operation.

Manufacturing improvement

Established the single source of truth that now underpins all production tracking, quality inspection, and management reporting.

Challenge

  • ERP data existed in isolation — no API, no programmatic access
  • All workflows manual, with no digital touchpoints beyond data entry
  • No common data layer for custom systems to build on
  • Reporting required manual extraction — delayed and error-prone

Solution

  • Structured SQL schema mirroring and extending the ERP data model
  • REST API endpoints enabling bidirectional communication with MIE Trak
  • Shared platform data layer consumed by all downstream systems
  • Single source of truth for production data across the organisation
Architecture
Implementation Notes

API-first design: each ERP entity — work orders, parts, routings, customers — mapped to structured SQL tables with appropriate relationships. Error handling, retry logic, and data validation built in from the start.

Key Decisions

Decoupled architecture

SQL layer survives ERP changes without rewriting consumer systems.

Schema-first design

All data models defined and reviewed before any application code was written.

Engineering Decision Record · EDR-01 · Data Architecture Pattern
Problem

How to make ERP data accessible to multiple custom systems without creating tight coupling to the ERP vendor's schema and API.

Options Considered

A. Direct ERP API calls from each app.
B. Shared SQL abstraction layer.
C. Message queue / event bus.

✓ Chosen · Option B

Shared SQL abstraction layer with REST API endpoints. All systems read/write through this layer — not directly to ERP.

Trade-offs & Lessons

The SQL layer adds a synchronisation step — ERP changes must propagate. In practice this was manageable and the benefit (full control over schema, query performance, and downstream API stability) far outweighed the cost. Data quality gaps in historical ERP data required additional normalisation logic not anticipated at the start.

Lessons Learned
What Worked

Building the SQL schema as an independent, well-documented layer made all subsequent systems faster to build. This investment paid back many times over.

Challenges Encountered

ERP data quality issues were more significant than anticipated — historical records used some fields inconsistently, requiring additional normalisation before the platform could rely on them.

Engineering Decision Record · EDR-02 · Mobile Platform Selection
Problem

Which platform to use for operator terminals — dedicated hardware, browser-based, or native mobile — given factory floor conditions and connectivity constraints.

Options Considered

A. Browser-based (any device).
B. Native Android application.
C. Dedicated industrial HMI terminals.

✓ Chosen · Option B

Native Android — ruggedised tablets with offline capability, familiar interaction model for operators, and full control over the user experience.

Trade-offs & Lessons

Native Android required handling intermittent factory floor connectivity — the application was revised to cache job data locally and sync when connected. Operator involvement during design prevented significant interface rework post-deployment. Connectivity handling was the single largest technical challenge and was underestimated at the outset.

Lessons Learned
What Worked

Involving operators in interface design during development — not post-launch — led to significantly higher adoption rates than typical enterprise software rollouts.

Challenges Encountered

Connectivity reliability on the factory floor required careful offline-first design. The application was substantially revised to handle intermittent network conditions gracefully.

Engineering Decision Record · EDR-03 · AI Integration Pattern
Problem

How to integrate AI validation in a way that engineers trust, use consistently, and that provides manufacturing-domain-relevant responses rather than generic LLM output.

Options Considered

A. Generic LLM API with minimal context.
B. Fine-tuned model on manufacturing data.
C. Structured prompting with domain context injection.

✓ Chosen · Option C

Structured prompting with manufacturing context injection — specific constraint sets, materials data, and process parameters provided per request.

Trade-offs & Lessons

Prompt engineering required significant iteration — early prompts produced overly generic responses. Domain-specific context injection was the key improvement. Framing the AI as an assistant (not a decision-maker) was critical to engineer adoption. Fine-tuning was considered but rejected due to cost and data volume requirements; structured prompting achieved the required quality with far lower overhead.

Lessons Learned
What Worked

Framing the tool as an assistant to engineers rather than a replacement led to consistent daily adoption. Engineers trusted outputs they could interrogate and override.

Challenges Encountered

Early prompt versions were too generic. Manufacturing-specific context injection required multiple iteration cycles before outputs were consistently useful for production decisions.

Engineering Decision Record · EDR-04 · Dashboard Architecture
Problem

How to deliver live management dashboards without building a separate BI platform — keeping the system within the existing infrastructure footprint.

Options Considered

A. Third-party BI tool (Power BI, Tableau).
B. Custom Python dashboard served from the platform.
C. Excel with live SQL connections.

✓ Chosen · Option B

Custom Python dashboards served from the existing platform — full control, no licence cost, consistent with the platform architecture already in place.

Trade-offs & Lessons

Building custom dashboards required more initial development time than configuring a BI tool, but produced a result that was precisely tailored to the manufacturing environment. Data completeness was an early challenge — dashboards highlighted gaps in historical data quality, requiring upstream data entry improvements before reporting could be fully trusted.

Lessons Learned
What Worked

Designing dashboards with management stakeholders present — rather than presenting finished views — meant the final product answered the questions they actually had.

Challenges Encountered

Data completeness was a challenge initially. Dashboards surface data quality gaps — which required upstream process improvements before reporting could be fully trusted.
